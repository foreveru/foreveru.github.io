robot framework -- write good test cases

发表于 2017-05-17 | 分类于 robotframework | 阅读次数

background

在team里做了几个模块的automation了,然而有时候schedule比较紧,写的时候更注重的是如何不漏掉每个observe point 和check point,以及如何构造keyword,却没怎么注意test case书写的规范，今天我们就来浅谈一下这方面的一些知识。

How to write good test cases using RobotFramework

以下参考了robotframework官方文档上的一些书写规范。

  1. test suite names

    • 尽量具有描述性
    • 没有拓展名，以空格代替下划线
    • 如果名字都是小写的，那么每个单词开头要大写
      例如:
      • login_tests.robot -> Login Tests
      • IP_v4_and_v6 -> IP v4 and v6

  2. test case names

    • 同样要具备可描述性
    • 如果一个suite包含了很多相似的测试用例，那么每个测试用例的名字可以短点儿，
      例如: invalid login文件里包含了相关的一些测试用例，那么下面这些名字就是比较合适的：
      * Test Cases *
      Empty Password
      Empty Username
      Invalid Username

      下面的就有点冗余了:

      Login With Empty Password Should Fail

  3. keyword names

    • 描述性强 & 清晰
    • 解释出keyword是完成什么功能的，[而不要用这个keyword来描述这个功能是怎么做的]
    • 具有概括性，
      例如： Inptut Text, Administrator logs into system
    • 当keyword name 短的时候：

      # 首字母大写
      like:
      Input Text
      Login With Valid Credentials

      # 而不要这样
      Input Valid Username And Valid Password And Click Login Button

  当keyword name长的时候(like a sentence)：
> \# 第一个单词的首字母大写
  Administrator logs into system

  1. naming setup and teardown

    • 描述功能的词，易懂
    • 如果setup or teardown包含很多步骤，那么可以
      • 使用更有概括性的词语
      • 使用robotframework 内置的keywork（如果这些步骤的关键字都已写好）: Run Keyword
        以下举例来展示好的写法与不好的写好

        Bad practice
        * Settings *
        Suite Setup Login To System, Add User, Activate Alarms And Check Balance

        Good practice
        * Settings *
        Suite Setup Initialize System

        * Settings *
        Suite Setup Run Keywords
        … Login To System AND
        … Add User AND
        … Activate Alarms AND
        … Check Balance
